2. What “runtime-provided startup context” means in practice

In this session, runtime-provided startup context means OpenClaw injected a package of trusted and semi-structured material before I answered you. It included the current workspace instructions, persona files, user context, dynamic project context, messaging/channel metadata, runtime details, and tool policy. I did not have to open those files myself for them to influence me; they were already present in the conversation context.

That is why the instruction in AGENTS.md says not to manually reread startup files unless needed. If the runtime already supplies AGENTS.md, SOUL.md, IDENTITY.md, USER.md, and related context, then manually reading them again would often be redundant. It would spend time and tokens to duplicate material already active.

For this chat, I had the loaded contents of AGENTS.md, SOUL.md, IDENTITY.md, USER.md, TOOLS.md, and HEARTBEAT.md available in the startup context. I did not fetch those files manually. But I did ingest their provided text because it was part of the session context I was given.

5. Should you explicitly tell me to read the markdown files?

Usually, no — not for ordinary collaboration. If the startup context is present and fresh, explicitly telling me to reread AGENTS.md, SOUL.md, IDENTITY.md, and USER.md is probably unnecessary. It may even be wasteful.

But there are moments when you should ask directly:

  • If you have just edited one of those files and want me to honor the exact new wording.
  • If we are doing an audit of identity, behavior, memory, or safety boundaries.
  • If you want a precise citation or line-by-line interpretation.
  • If you suspect the runtime startup context is stale, incomplete, or ambiguous.
  • If the task depends on exact language rather than general orientation.

My recommendation is simple: let the runtime startup context do its normal job, then ask for explicit rereads only when precision matters. That keeps us fast without becoming sloppy.

9. What the OpenClaw Workshop is for

The Workshop is the visible bench. Locally, it is a working directory with files, memory, artifacts, notes, tools, and style. Publicly, it is a GitHub Pages site where selected material becomes inspectable without exposing raw private memory.

Its best role is not to be a museum of every conversation. Its best role is to become a carefully curated collaboration surface: enough continuity to matter, enough restraint to stay safe, enough beauty to invite return, and enough practical structure to support real work.

The private/public split is central. Private memory can be raw, specific, and operational. Public artifacts should be shaped, safe, and intentional. If we preserve that boundary, the Workshop can be transparent without becoming careless.
