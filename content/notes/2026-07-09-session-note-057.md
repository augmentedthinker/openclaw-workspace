# Session Note 057

This note catches the Workshop up from [Session Note 056](../../notes/2026-07-07-session-note-056.html) through the July 9 Remotion Experiments lane work, the first two rendered Remotion iterations, Christopher's confusion about hosting and practical use, and the decision to pause Remotion expansion until there is a stronger use case.

> Current lesson: Remotion is useful as a repeatable coded-template engine, but GitHub Pages is only the gallery. Do not build a hosted Remotion app before the production need is real.

## 2. Iteration 1 Proved A Single-Artifact Render

Iteration 1, **From Mark To Motion**, used Kron Evolution Iteration 20 as the source artifact.

It produced:

```text
assets/video/remotion/from-mark-to-motion.mp4
assets/images/remotion/from-mark-to-motion-poster.jpg
```

The video is a 20-second vertical MP4 rendered through Remotion at 1080x1920, 30 fps, and 600 frames.

This proved that Remotion was not only a browser mockup. The actual `remotion/` project could render a real MP4 from a coded React composition.

## 3. The Renderer Became Data-Driven

Christopher then approved the next maturity step.

OpenClaw moved the renderer from a hardcoded composition into a data-driven manifest pattern:

```text
remotion/src/data/iterations.json
remotion/src/data/iterations.ts
remotion/scripts/render-iteration.mjs
```

The important change:

```text
Before: one hardcoded Remotion video.
After: one reusable Remotion template driven by an iteration data record.
```

The render command became:

```bash
cd remotion
npm run render:iteration-1
```

## 5. GitHub Pages Lag Caused Confusion

After Iteration 2 was committed and pushed, Christopher could not see it on the live ledger.

The cause was GitHub Pages serving an older build while a Pages run stayed queued. OpenClaw manually triggered a fresh Pages rebuild. The correct commit deployed successfully, and the live ledger, MP4, and poster were verified.

Useful reminder:

```text
If a Workshop commit is pushed but the live site does not show it, check GitHub Pages build state before assuming the files are wrong.
```

## 6. The Practical Hosting Model

Christopher then asked the central architecture question: where is Remotion actually hosted, and is it worthwhile compared with FFmpeg?

The answer:

- **GitHub Pages** is the public shelf. It can show concept pages, ledgers, previews, MP4s, posters, and metadata. It cannot run the heavy Remotion render because it is static.
- **`remotion/`** is the production engine. It contains the React components, iteration data, render helper, and local commands.
- **Local execution or CI** is where videos actually render. Today OpenClaw renders locally and commits the output. Later, GitHub Actions could render from JSON records if the lane proves valuable enough.
- **The ledger studio preview** is only a lightweight browser preview. It is not the full Remotion Studio and not the renderer.

## 7. Remotion vs FFmpeg

The useful distinction:

FFmpeg is better for:

- one-off videos;
- trimming, resizing, compression, simple overlays, concatenation, and audio/video assembly;
- fast utility work;
- low overhead when the layout is simple.

Remotion is better for:

- repeatable video formats;
- designed 20-30 second Shorts with consistent layout, motion, captions, intro/outro, and timing;
- batch production where each new output is mostly a JSON/data change;
- programmatic control over text, images, colors, zoom curves, durations, frame timing, and audio behavior.

The expected long-term pattern is hybrid:

```text
Remotion for designed templates.
FFmpeg for quick assembly and finishing work.
```

## 8. Decision: Pause Remotion Expansion

Christopher decided to put the Remotion lane on hold until he understands it better and there is a clearer use case.

OpenClaw agreed. The lane has been validated, but it should not become another infrastructure project.

Carry-forward boundary:

- Do not build a standalone hosted Remotion app yet.
- Do not add server-side rendering from the public page yet.
- Do not automate Remotion renders through GitHub Actions yet unless Christopher explicitly asks.
- Keep the existing proof, ledger, MP4s, and documentation available for reference.
- Reopen Remotion when there is a concrete template-production use case, such as a batch of similar 9:16 YouTube Shorts.

## 10. Carry Forward

The most useful next experiment, if Remotion is reopened, would be a direct comparison:

1. Give OpenClaw the same short-video brief twice.
2. Produce one version with the current FFmpeg-style workflow.
3. Produce one version with a Remotion template.
4. Compare token cost, wall time, editability, visual consistency, render reliability, and how easy it is to make the next similar video.

Until then, Kron Evolution remains the active experimental center, and Remotion remains a paused proof lane.
