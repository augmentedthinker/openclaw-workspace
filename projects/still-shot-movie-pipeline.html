Still-Shot Movie Pipeline

Status: prototype proven First output: 10-second 1080p MP4 Source: OpenAI research brief + OpenClaw prototype

This project turns static images into short, storyboarded videos for the AugmentedThinker YouTube signal lane.

The idea came from Christopher's Gemini experiment: a small set of still images became movie-like through slow pans, zooms, and shot sequencing. OpenClaw then created a first 10-second prototype from existing Workshop and Bluesky-adjacent assets.

View the first still-shot movie prototype artifact.

Strategic read

The useful lesson from the research brief is not that we should immediately build a full browser-native video editor. The useful lesson is that there are clear levels of complexity.

  1. Local native rendering now: use ffmpeg or similar local tools to turn stills into MP4s with pan, zoom, timing, captions, and audio.
  2. Scripted composition next: represent videos as simple timelines: scenes, image paths, motion type, duration, caption, audio, output format.
  3. Browser preview later: use canvas, WebGL, and WebCodecs only if repeated production makes live preview worth the engineering.
  4. Cloud AI escalation only when justified: use image-to-video or diffusion APIs for shots that cannot be achieved through still-image motion, and track cost before scaling.

For OpenClaw, the right near-term lane is scripted composition, not a full custom video application.

What we already proved

  • OpenClaw can create a short still-shot movie locally from existing assets.
  • The first video uses four 2.5-second image shots with push-in, pull-back, and lateral drift motion.
  • The first outputs are a 10-second, 1920x1080, 30 fps H.264 MP4 and a matching 1080x1920 vertical Shorts MP4.
  • A local npm-provided ffmpeg binary produced the first render, and system ffmpeg is now installed for normal OpenClaw skill use.
  • The video-frames skill successfully extracted a 1080x1920 JPEG frame from the vertical Shorts prototype.
  • The finished clip can be embedded on a public Workshop page.

Current tooling status: as of 2026-05-25, system ffmpeg is installed at /usr/bin/ffmpeg, and OpenClaw reports video-frames as ready and visible to the model.

Smoke test: OpenClaw extracted /tmp/openclaw-video-frame-check.jpg from the vertical Shorts render at 1080x1920, confirming the skill can inspect rendered videos.

Operating pipeline

  1. Pick a theme: agent field note, signal loop, product idea, or visual micro-story.
  2. Select 3 to 6 still images from Workshop assets or generate new ones.
  3. Write a tiny storyboard: opening image, middle tension, closing idea.
  4. Assign motion to each shot: push in, pull back, pan left or right, hold, or reveal.
  5. Add optional captions, title card, voiceover, music, or ambient bed.
  6. Render MP4 locally.
  7. Review before upload.
  8. Publish only after Christopher approval.
  9. Log the URL and signal.

Technical direction

  • Near term: keep using native ffmpeg; use video-frames for frame checks and thumbnail extraction; create reusable scripts around a simple JSON timeline; generate both horizontal 16:9 prototypes and vertical 9:16 Shorts versions; add title and caption support before chasing advanced effects.
  • Medium term: consider MoviePy only if Python timeline editing becomes easier than direct ffmpeg; consider Revideo or canvas-based rendering if we need live preview, reusable visual templates, or kinetic typography.
  • Later: explore WebCodecs plus a muxer such as Mediabunny if we build an interactive local web tool; explore AI image-to-video APIs for selected high-value shots, not as the default engine.

Research takeaways worth keeping

  • Canvas-centric rendering is better than DOM screenshot pipelines when performance matters.
  • ffmpeg.wasm is probably the wrong primary path for Chromebook-class hardware because of speed, memory, and browser isolation overhead.
  • WebCodecs is the most interesting browser-native path for future hardware-accelerated encoding.
  • Generative image-to-video models belong in a hybrid cloud layer because full-quality AI video generation is compute-heavy and cost-sensitive.
  • The practical first product is not AI video generation. It is reliable storyboarding, motion, captions, voiceover, and repeatable publishing.

Boundary: uploads, public metadata, comments, and channel changes remain approval-gated unless Christopher explicitly defines a narrower autonomous YouTube routine.
